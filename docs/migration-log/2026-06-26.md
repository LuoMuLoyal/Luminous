# Migration Log - 2026-06-26

## Notification module (backend + frontend)

Implemented the full user notification system across Lucent backend and Luminous frontend, replacing the deferred "notification bell inbox" placeholder with real data.

### What changed

**Lucent backend**

- Added `UserNotificationType` enum and `UserNotification` model to Prisma schema with `userId` relation, `isRead`/`readAt` state, `action`/`actionPayload` routing fields, and standard audit timestamps.
- Applied migration `20260626000000_add_user_notifications` to dev PostgreSQL.
- Created `NotificationsModule` with standard NestJS controller/service/DTO structure:
  - `GET /api/v1/notifications` — paginated list with `UserNotificationListItemDto`.
  - `GET /api/v1/notifications/unread-count` — `UnreadCountResponseDto`.
  - `GET /api/v1/notifications/:id` — `UserNotificationDetailDto` (auto-marks as read on fetch).
  - `POST /api/v1/notifications` — manual creation for testing.
  - `PATCH /api/v1/notifications/:id/read` and `/:id/unread` — toggle read state.
  - `PATCH /api/v1/notifications/mark-all-read` — batch mark all as read.
  - `DELETE /api/v1/notifications/:id` — soft delete.
- Wired `NotificationsModule` into `AppModule` and into consuming modules (`TodayAnalysisModule`, `DataExportModule`, `AuthModule`) so `NotificationsService` can be injected.
- Added inline notification generation at three business events:
  - `TodayAnalysisService` — creates `ai_today_summary` after successful AI stream/generate.
  - `DataExportService` — creates `report_generated` after successful export.
  - `AuthService` — creates `password_changed` after `changePassword` or `setPassword`.
- Exported updated `docs/openapi.json` (68 paths / 178 schemas).

**Luminous frontend**

- Regenerated `packages/lucent_openapi` from the updated OpenAPI contract; new DTOs include `NotificationListItemDto`, `NotificationDetailDto`, `UnreadCountResponseDto`.
- Added router entries: `/notifications` and `/notifications/:id` with `_slidePage` transition.
- Added `notificationsApi` getter to `LucentDioClient` and `lucentNotificationsApiProvider` to network providers.
- Created `notification_providers.dart` with:
  - `notificationUnreadCountProvider` — `FutureProvider` for badge count.
  - `notificationListPageProvider` — `FutureProvider` for paginated list.
  - `notificationDetailProvider` — `FutureProvider.family` for detail by ID.
  - `NotificationListController` (`AsyncNotifier`) with `markAllAsRead` and `deleteNotification` mutations plus auto-invalidation.
- Created `NotificationListPage`:
  - Grouped by 今天 / 昨天 / 更早 with section headers.
  - App-bar "全部已读" batch action.
  - `Dismissible` swipe-to-delete with confirm dialog.
  - `AppStateErrorView` / `AppSkeletonShimmer` + `AppInlineSkeletonBlock` for loading/error states.
  - Empty state with localized copy.
- Created `NotificationDetailPage`:
  - Type chip, content card, relative timestamp, action bar.
  - "去处理" button routes via `switch(action)` to `today`, `report`, `/account` using `GoRouter.push`.
  - "设为未读" and "删除" actions with confirmation dialogs.
- Created `NotificationListItem` widget:
  - Unread blue dot indicator.
  - Relative time (HH:mm for today, MM-DD for older).
  - Truncated title + body preview.
- Wired Today top-bar `_NotificationButton` and Mine page notification row to `context.push('/notifications')`.
- Added ~20 ARB keys to `app_zh.arb`/`app_en.arb` and ran `flutter gen-l10n`.

### Why

The notification bell was a visible placeholder in both Today and Mine pages. Building the real backend-to-frontend notification loop completes a core v4.0.0 user expectation: users receive explicit feedback when AI summaries finish, reports are exported, or security events (password changes) occur, and they can act on those events directly from the inbox.
