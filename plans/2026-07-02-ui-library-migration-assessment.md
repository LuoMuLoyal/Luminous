# Luminous 前端 UI 迁移评估（自定义体系 -> 第三方组件库）

## 1. 文档目的

这份文档用于回答一个明确问题：

> 在 **Luminous** 由单人结合 AI 维护、主精力需要转向后端药品/食品数据链路的前提下，前端是否应该从当前自定义 UI 体系迁移到第三方组件库。

本文聚焦三件事：

- 受影响范围
- 收益与成本
- 是否值得做，以及推荐选型

这不是最终实施方案，也不是当前状态文档。它是执行前的决策依据。

---

## 2. 当前约束与真实问题

### 2.1 维护现实

- 当前项目长期由 **1 人 + AI** 维护。
- 主精力正在被 Lucent/Luminous 相关的药品中文库、英文库映射关系、食品数据处理链路占用。
- 前端不是当前最想深耕的方向，目标更偏向：
  - 降低 UI 基础设施维护成本
  - 降低 AI 写偏、写散、写重复的概率
  - 提高新增页面的默认一致性

### 2.2 当前 UI 体系的实际痛点

- 自定义设计体系经历过多轮修改后形成，历史包袱已经存在。
- AI 会出现以下典型问题：
  - 明明已有共享组件，仍重新手写一个近似版本
  - 明明已有 token，仍直接写颜色、圆角、边距
  - 同一类型页面之间视觉风格不稳定
  - 弹窗、sheet、表单、卡片、头部结构反复出现近似但不一致的实现
- 由于开发节奏快，新增页面经常在没有充分模拟器复看的情况下就进入下一个 feature，导致视觉偏差滞后暴露。

### 2.3 代码侧已观察到的耦合面

以 2026-07-02 的代码搜索结果估算：

- `AppTypographyTokens / AppThemeSurface / AppSpacingTokens / AppRadiusTokens / PageScaffoldShell / PageSectionCard / AppSettingRow / AppSettingsSection`
  - 总引用次数约：`2158`
  - 命中的文件数约：`174`
- 直接 import `core/theme`、`core/design`、`core/widgets` 的文件数约：`183`

这说明当前体系不是“几个工具组件”，而是已经深度渗透到各 feature 的 presentation 层。

### 2.4 已存在的重复/分叉信号

代码里已能看到多套弹窗写法并存：

- 共享封装：`lib/core/widgets/common/app_dialog.dart`
- 直接 `AlertDialog`
- 直接 `Dialog`
- 直接 `showModalBottomSheet`

这类分叉说明问题不只是“token 不够”，而是 **约束不够强，入口不够单一**。

---

## 3. 选项对比

### 选项 A：继续维护当前自定义体系

含义：

- 保留 `core/theme`、`core/design`、`core/widgets`
- 继续补规范、补封装、补测试、补 review 约束
- 不迁移到第三方组件库主导

### 选项 B：迁移到第三方组件库主导

含义：

- 第三方组件库接管主题、基础控件、overlay、表单 primitives、部分导航 primitives
- 基础 primitives 允许直接进入业务页面，不再强制保留一层通用 `AppButton / AppInput / AppSelect` 包装壳
- 本地只保留业务语义门面和少量业务复合组件
- 逐步冻结并淘汰当前自定义基础 UI

---

## 4. 结论

在当前维护条件下，**更推荐选项 B：迁移到第三方组件库主导**。

原因不是“第三方一定更漂亮”，而是：

- 它更适合单人结合 AI 的维护模型
- 它更容易建立硬边界
- 它更能减少“AI 写出另一套近似组件”的自由度
- 它更符合“把精力让给后端和数据链路”的优先级
- 在已经接受页面级重构的前提下，直接使用第三方 primitives 比再包一层通用 wrapper 更直接，也更少重复建设

如果继续维护当前自定义体系，也不是做不到，但需要投入一轮接近“重建内部组件库治理”的工作。对当前人力结构来说，这条路的边际收益不高。

---

## 5. 推荐组件库

当前更推荐：**Forui**

推荐理由：

- 已验证与当前本地环境版本兼容：`Flutter 3.44.0`
- 具备较完整的主题、基础组件、交互组件、responsive 能力
- 适合接管 `dialog / sheet / popover / form / tabs / select / scaffold primitives`
- 文档和体系化程度足够支撑“作为主 UI 基座”尝试

这里的推荐前提是：

- 目标是让第三方库直接成为基础 UI 层
- 目标不是构建一套“未来可无痛切换任意组件库”的通用抽象层
- 允许业务页面随着迁移逐步改写成新库风格

不推荐本次迁移优先选择 `shadcn_ui` 作为主库，原因是它当前更像“可用但仍在持续演进”的体系，稳定性预期不如本次迁移目标需要的那样明确。

## 6. 迁移的受影响范围

## 6.1 必然受影响的目录

### 主题与设计 token

- `lib/core/theme/`
- `lib/core/design/`

受影响原因：

- 当前颜色、排版、圆角、间距、surface 扩展都在这里
- 如果第三方组件库要全面接管，这一层至少要从“主来源”降级为“兼容层”或“迁移桥接层”

### 共享基础组件

- `lib/core/widgets/common/`
- `lib/core/widgets/layout/`
- `lib/core/widgets/settings/`

受影响原因：

- 当前页面头部、卡片、错误态、按钮、设置行、弹窗、布局容器都在这里
- 这层正是未来最应被统一入口替换的部分

### 主要 feature 的 presentation 层

- `lib/features/today/presentation/`
- `lib/features/record/presentation/`
- `lib/features/medicine/presentation/`
- `lib/features/report/presentation/`
- `lib/features/mine/presentation/`
- `lib/features/settings/presentation/`
- `lib/features/auth/presentation/`
- `lib/features/search/presentation/`
- `lib/features/assistant/presentation/`
- `lib/features/scan/presentation/`
- `lib/features/notification/presentation/`
- `lib/features/shell/presentation/`

受影响原因：

- 这些页面大量直接依赖当前 token 和共享组件
- 真正的 UI 一致性问题基本都发生在这一层，而不是 data/domain 层

## 6.2 影响较小或基本不受影响的目录

- `lib/features/*/data/`
- `lib/features/*/domain/`
- `packages/lucent_openapi/`
- 网络、通知、路由、数据源处理链路

说明：

- 迁移目标主要是前端表现层
- 后端 API、实体模型、仓库接口、OpenAPI 生成客户端不应成为这次迁移的主战场

## 6.3 会被重点替换的能力面

- 主题定义
- 文本样式入口
- 页面 scaffold / section / card
- 对话框、底部 sheet、popover
- 表单输入、select、开关、tabs
- 空态/错误态的通用视觉容器
- 设置页行组件
- 部分按钮、chip、badge、list item

## 6.4 建议保留本地封装的能力面

迁移到第三方组件库后，不建议再保留一层普遍适用的基础组件别名壳（例如 `AppButton`、`AppInput`、`AppSwitch` 只是简单转发到第三方组件）。以下能力仍建议保留本地门面：

- 业务卡片
- 医疗/药品/记录场景的专用复合组件
- 统一的鉴权提示入口
- 统一的删除确认入口
- 统一的错误态/加载态入口
- 页面级 scaffold / header / section 这类与当前产品结构强绑定的骨架组件
- 与 L10n、路由、权限、业务动作绑定的组件

原因：

- 这些不是纯 UI primitives，而是业务约束的承载点
- 这些封装承担的是产品语义，而不是单纯替第三方组件改名
- 如果只是把第三方组件透传一层再改名，维护成本会上升，但可替换性收益并不高

---

## 7. 收益

## 7.1 直接收益

### 统一性收益

- 页面不再主要依赖 AI 自由拼装 `Container + Text + Padding + BorderRadius`
- 风格统一更依赖库的默认约束，而不是每个页面都记得正确使用 token
- 新页面默认更接近可接受结果

### 维护收益

- 基础控件的视觉与交互维护从“自定义体系持续修补”转移为“升级/适配第三方库”
- 单人维护时，可显著减少在弹窗、sheet、表单、卡片等通用控件上的重复劳动

### AI 协作收益

- 可以把指令收紧为“基础 UI 优先直接使用 Forui，不允许继续手写旧基础组件，不允许新增通用别名壳”
- AI 的选择空间变小，结果更稳定
- 对新页面的 prompt 约束更明确，减少“明明有封装却没用”的情况

### 视觉回归收益

- 第三方库的 primitives 往往已有较成熟的间距、状态、交互反馈
- 可以减少“单个页面看着还行，放到整体里不协调”的问题

## 7.2 间接收益

- 让主要注意力回到后端和数据链路
- 降低未来继续扩页面时的风格漂移
- 让前端演进更偏“组合已有能力”，而不是“每次重新设计一个页面结构”

---

## 8. 成本

## 8.1 一次性迁移成本

### 技术成本

- 需要建立新的主题桥接方式
- 需要替换大量共享组件和页面局部结构
- 需要处理第三方库与现有 Material / Riverpod / GoRouter 结构的接缝

### 代码改动成本

- 不是局部替换，至少会动到 `core/theme`、`core/design`、`core/widgets` 和大量 feature presentation 文件
- 若按“全面接管”目标执行，改动面会比较广，短期 diff 体量较大

### 验证成本

- 需要大量实际页面回归
- 需要重新检查移动端和桌面/大屏下的布局表现
- 若缺少 golden/widget 基线，视觉回归验证仍然需要真实运行页面

## 8.2 迁移期成本

- 一段时间内会出现双体系共存
- 新功能开发速度会下降
- 需要先处理基础设施，再继续高频堆 feature

## 8.3 未来维护成本

- 需要跟踪第三方库版本变化
- 需要维护少量业务语义封装，避免把鉴权、删除确认、错误态、页面骨架等能力打散到页面里
- 需要在 AI/AGENTS 规则里新增明确限制，否则仍会回到“第三方 primitives + 老自定义组件 + 原生 Material”混用状态

---

## 9. 风险

## 9.1 最大风险不是迁移失败，而是迁到一半

半迁移状态会导致：

- 一半页面用老体系
- 一半页面用新体系
- token、共享组件、主题入口继续双轨并存
- AI 更难判断该用哪套

这会比现在更乱。

## 9.2 第三方库不是“自动稳定”

第三方库只能提升基础控件层的稳定性，不能自动解决：

- 业务组件层不一致
- 页面信息层级差
- 不跑模拟器、不做视觉回归
- AI 越权直接手写 UI

所以迁移后仍需配套治理。

## 9.3 组件库升级与 API 演进风险

- Forui 仍不是“几十年不动”的稳定平台
- 若未来 API 演进较快，直接在业务页面大量使用第三方组件会增加升级痛苦

但在当前目标下，这个风险是被接受的。因为当前更看重的是尽快替换掉手写基础 UI，而不是优先优化“未来换库成本”。

---

## 10. 如果不迁移，继续自定义需要补的治理项

如果决定不迁移，至少要补齐以下机制，否则当前痛点还会继续出现：

- 统一 UI 入口目录，删除重复组件
- 禁止 feature 页面直接手写基础弹窗、sheet、卡片、设置行
- 禁止页面直接写颜色、圆角、边距
- 建立组件 catalog/demo 页
- 为关键共享组件和关键页面增加 widget/golden 验证
- 在 AGENTS 里写清楚允许使用的组件入口
- 对 AI 输出增加检查规则，发现重复 UI 结构必须回收为共享组件

问题在于：

- 这本质上是在继续投资一套内部组件库治理体系
- 对当前“前端不是主要精力投入方向”的现实并不友好

因此，不迁移不是更省事，只是把成本换一种形式继续支付。

---

## 11. 推荐决策

### 推荐结论

**建议启动迁移到 Forui 主导的 UI 基础设施改造。**

### 推荐边界

不是“所有东西都交给第三方”，而是：

- 第三方库接管：
  - theme
  - primitives
  - dialog/sheet/popover
  - form controls
  - tabs/select/switch/button 等基础交互组件
- 本地保留：
  - 业务复合组件
  - 路由与鉴权绑定组件
  - 删除确认、鉴权提示、错误态等统一业务门面
  - 页面骨架与少量高频复合布局

### 明确不做的事

- 不建设一套普遍适用的 `AppButton / AppInput / AppSwitch / AppDialog` 通用基础别名层
- 不为了“也许未来换库”而提前做过度抽象
- 不继续给旧自定义基础组件补新能力

### 不推荐的做法

- 不做全仓一次性大爆改
- 不允许未来 feature 继续混写“第三方原始组件 + 老自定义组件 + 原生 Material”
- 不允许业务页面自由手写基础视觉样式
- 不允许新增“仅仅转发第三方 API”的基础组件包装壳

---

## 12. 对你当前情况最有价值的原因

这次迁移最核心的价值，不是“更好看”，而是：

- 让前端更像一个可复用平台，而不是一堆需要持续盯视觉细节的页面
- 把你的注意力从“为什么这个新页面又写歪了”转移到“后端药品/食品链路是否正确”
- 让 AI 在前端的发挥空间更收敛，减少维护者本人做 UI 审美和一致性兜底的频率
